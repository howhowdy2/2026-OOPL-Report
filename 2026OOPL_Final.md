## 專案簡介

### 遊戲簡介

本專案復刻休閒卡牌經營遊戲《Stacklands》（堆疊大陸）。玩家透過拖曳、堆疊卡牌來建造村莊、收集資源、製作工具與裝備，並確保村民在每月結算時獲得足夠食物以存活。遊戲目標是收集資源、打造裝備、訓練村民，最終擊敗最終 Boss 贏得勝利。

本作品的核心玩法包含卡牌拖曳與堆疊、資源採集與合成、卡包購買與經濟系統、村民職業轉職、裝備穿戴、動物飼養、怪物戰鬥、月結算食物消耗、存檔讀檔，以及作弊選單等功能。

## 遊戲介紹

### 遊戲規則

玩家使用滑鼠拖曳卡牌進行遊戲操作。主要目標是管理村莊資源與村民，最終擊敗 Boss。

操作方式如下：

| 操作 | 功能 |
|------|------|
| 滑鼠左鍵拖曳 | 拖曳卡牌、堆疊卡牌 |
| 滑鼠右鍵拖曳 / WASD | 平移鏡頭 |
| 滑鼠滾輪 | 縮放畫面（0.5x ~ 4.0x） |
| 播放按鈕 | 正常速度 |
| 快進按鈕 | 2 倍速 |
| 暫停按鈕 | 暫停遊戲時間 |
| F1 | 開啟 / 關閉作弊選單 |

遊戲主要規則：

- 將卡牌堆疊在一起可觸發合成、採集、戰鬥等行為。
- 每個月結算時，村民會消耗食物。若食物不足，村民會餓死。
- 當所有村民死亡時，遊戲結束（Game Over）。
- 場上卡牌數量超過當前最大值時，月結算會觸發卡牌溢出警告，遊戲暫停直到玩家賣出多餘卡牌。
- 將卡牌拖至賣出格（SellSlot）可賣出獲得金幣。
- 使用金幣可購買不同等級的卡包，開啟卡包獲得新卡牌。

## 程式設計

### 程式架構

本專案採用物件導向方式設計，將遊戲中的不同功能拆分為獨立類別，以降低耦合並方便維護。

主要類別如下：

| 類別 | 功能 |
|------|------|
| App | 控制整體遊戲流程與狀態切換（START、MAIN_MENU、GAME_INIT、UPDATE、GAME_OVER、END） |
| CardManager | 管理所有卡牌的生成、移除、拖曳、碰撞偵測與堆疊邏輯 |
| CardFactory | 卡牌工廠，透過註冊表根據類型名稱建立對應卡牌物件 |
| EventManager | 管理月份時間、遊戲速度（正常/快進/暫停）、鏡頭平移與縮放 |
| RecipeManager | 管理合成配方與職業轉職配方 |
| TaskScheduler | 集中管理採集、合成、戰鬥等排程任務 |
| UIManager | 管理 HUD、按鈕、時間條、暫停選單等 UI 元素 |
| CheatMenu | 作弊選單，按 F1 開啟，可生成任意卡牌 |
| Card | 所有卡牌的基底類別 |
| CombatCard | 戰鬥卡牌基底，包含生命、攻擊、防禦、裝備欄 |
| CharacterCard | 村民卡牌，繼承 CombatCard，具有食物消耗 |
| WanderingCard | 會移動的卡牌基底，提供漫遊與追蹤行為 |
| AnimalCard | 動物卡牌，繼承 WanderingCard，具有特殊能力（如生產羊毛、雞蛋） |
| MonsterCard | 怪物卡牌，繼承 WanderingCard，會追蹤並攻擊村民 |
| BuildingCard | 建築卡牌 |
| ResourceCard | 資源卡牌 |
| FoodCard | 食物卡牌，具有營養值 |
| EquipmentCard | 裝備卡牌，提供攻擊力、防禦力、血量等加成 |
| StructureCard | 可採集資源的結構物（如礦場、森林） |
| LocationCard | 探索地點，可重複探索獲取掉落物 |
| CardPack | 卡包，開啟後隨機生成卡牌 |
| CoinCard | 金幣卡牌 |
| SellSlot | 賣出格，拖入卡牌可賣出換取金幣 |
| BlankSlot | 購買格，放入金幣可購買對應卡包 |
| CombatArena | 戰鬥場地，管理戰鬥雙方、攻擊流程與動畫 |
| FloatingTextManager | 浮動文字管理，顯示傷害數字、治療、狀態效果 |
| Hotpot | 火鍋，自動吸收堆疊的食物卡牌 |
| CoinChest | 金幣箱，可儲存最多 100 金幣 |
| ResourceChest | 資源箱，可儲存多張資源卡牌 |
| MagicGlue | 魔法膠水，黏住整疊卡牌使其無法拖動 |

遊戲主流程由 `App` 控制，並使用 `State` 列舉切換不同狀態。各子系統（`CardManager`、`UIManager`、`EventManager`）由 `App` 以 `std::unique_ptr` 持有，負責各自領域的更新與管理。

### 程式技術

#### 1. 物件導向設計

本專案採用物件導向方式設計，將遊戲中的卡牌、系統與 UI 拆分成獨立類別。

在繼承與多型方面，所有卡牌繼承自 `Card` 基底類別，透過虛擬函式 `OnStacked()`、`Update()`、`OnMonthEnd()`、`CanDrag()`、`CanStackOnto()`、`IsSticky()` 等實現多型行為。例如 `CharacterCard` 在月結算時消耗食物、`MonsterCard` 在更新時追蹤村民、`Hotpot` 在堆疊時自動吸收食物卡。不同卡牌雖然行為各異，但都可透過 `Card` 的共同介面統一管理。

戰鬥相關卡牌透過 `CombatCard` 提供共同的戰鬥介面，包含 `TakeDamage()`、`GetAttack()`、`GetDefense()`、`IsStunned()`、`ApplyStun()` 等。`CharacterCard`、`AnimalCard`、`MonsterCard` 都繼承此介面，讓戰鬥系統無需關心具體卡牌類型。

會移動的卡牌透過 `WanderingCard` 抽象基底類別實現共同的漫遊與追蹤行為，`AnimalCard` 與 `MonsterCard` 各自覆寫移動邏輯。

在封裝方面，每張卡牌將自己的狀態與操作封裝在類別內。例如 `Card` 封裝位置、縮放、Z 軸排序、堆疊關係（`m_CardBelow` / `m_CardAbove`）；`CombatCard` 封裝裝備欄位與戰鬥數值；`EventManager` 封裝月份計時、遊戲速度與鏡頭狀態。

在組合關係方面，`App` 作為主控制器持有 `CardManager`、`UIManager`、`EventManager` 等子系統。`CombatCard` 內含 `EquipSlotData` 陣列管理裝備欄。`CombatArena` 組合戰鬥雙方的卡牌參考與浮動文字管理器。

#### 2. 智慧指標與記憶體管理

專案大量使用 C++ 智慧指標管理物件生命週期：

- `std::unique_ptr`：用於 `App` 持有子系統（`CardManager`、`UIManager`、`EventManager`）、`TaskScheduler` 持有排程任務、`CombatArena` 與 `TimeBar` 等獨佔所有權物件。
- `std::shared_ptr`：用於 `CardManager` 管理場上所有卡牌，讓多處程式碼可安全參考同一張卡牌。
- `std::weak_ptr`：用於 `TaskScheduler` 中的任務對卡牌的弱參考，避免循環參考導致記憶體洩漏。

場景切換與卡牌移除時，透過智慧指標自動釋放資源，避免手動管理記憶體的錯誤。

#### 3. 工廠模式（Factory Pattern）

`CardFactory` 使用註冊表模式，透過模板函式將各種卡牌類型的建構函式註冊到 map 中。建立卡牌時只需傳入類型名稱字串，工廠即可查表建立對應的卡牌物件，符合開放封閉原則（OCP）。

#### 4. 資料驅動設計（Data-Driven Design）

遊戲的卡牌、卡包、配方與職業資料皆透過 JSON 設定檔管理：

| 設定檔 | 內容 |
|--------|------|
| Cards.json | 卡牌資料庫：名稱、類型、售價、圖示、血量、攻擊、防禦、食物消耗、營養值、描述、效果、裝備欄 |
| Packs.json | 卡包定義：卡包名稱、售價、卡牌數量、卡池與權重 |
| Recipe.json | 合成配方：輸入材料與數量 → 輸出卡牌與所需時間 |
| Profession.json | 職業轉職：裝備組合 → 對應職業 |

這讓數值調整與內容新增不需修改 C++ 程式碼，只需編輯 JSON 檔案即可。

#### 5. 狀態機設計

遊戲使用狀態機管理不同畫面與流程：

```
START → MAIN_MENU → GAME_INIT → UPDATE → GAME_OVER → MAIN_MENU
                                  ↑                      ↓
                                  └──────────────────────┘
```

`App` 根據 `State` 列舉決定執行哪個階段的邏輯。`EventManager` 另外管理遊戲速度狀態（`NORMAL`、`FAST`、`PAUSE`），透過 `GetScaledDtMs()` 讓所有時間相關邏輯自動配合暫停與加速。

#### 6. 卡牌堆疊系統

卡牌之間透過 `m_CardBelow` 與 `m_CardAbove` 指標形成雙向鏈結結構。堆疊時呼叫虛擬函式 `OnStacked()` 判斷是否可堆疊，並觸發對應的合成、採集或戰鬥行為。特殊卡牌如 `MagicGlue` 透過 `IsSticky()` 機制讓整疊卡牌無法被拖動。

#### 7. 戰鬥系統

戰鬥系統由 `CombatArena` 管理，支援以下機制：

- 攻擊頻率由攻擊速度決定
- 命中率判定（Hit Chance）
- 防禦值減傷
- 裝備加成自動套用
- 多種狀態效果：暈眩（Stun）、流血（Bleed DoT）、中毒（Poison DoT）、狂暴（Frenzy，2 倍攻速）、無敵（Invulnerable）、暴擊（2 倍傷害）、吸血（Lifesteal）、治療（Heal）
- `FloatingTextManager` 顯示浮動傷害數字與狀態文字
- `AttackResolver` 負責傷害計算與效果觸發

怪物具有 AI 行為：平時在場上漫遊，接近村民時會自動追蹤並發起戰鬥。

#### 8. 月結算系統

`EventManager` 追蹤月份時間（預設每月 120 秒）。月結算時觸發以下事件：

- 檢查每位村民的食物消耗，從火鍋或場上食物卡扣除
- 食物不足時村民死亡
- 所有村民死亡時觸發 Game Over
- 場上卡牌超過當前最大值時顯示溢出警告
- 建築物觸發生產（如農場產出食物）

#### 9. 經濟系統

- `SellSlot`：拖入卡牌自動賣出，金幣堆疊在賣出格上
- `BlankSlot`：7 個購買格，對應不同等級卡包（從「Humble Beginnings」到「Order and Structure」）
- `CoinChest`：金幣箱，最多儲存 100 枚金幣
- 卡包開啟使用加權隨機掉落，第二次開包保證包含一名村民

#### 10. 動物特殊能力

`AnimalCard` 透過 `ProcessSpecialAbility()` 定期觸發特殊能力（如牛生產牛奶、雞生產雞蛋），並透過 `ISpawnListener` 介面回呼 `CardManager` 處理新卡牌生成。

#### 11. 存檔與讀檔

遊戲支援 JSON 格式的存讀檔功能（使用 nlohmann/json 函式庫）。存檔內容包含：

- 所有場上卡牌的位置、狀態、血量、裝備
- 目前月份與時間進度
- 鏡頭縮放與位置
- 購買格的金幣狀態
- 排程任務（採集、合成、戰鬥進度）

Game Over 時存檔會被刪除，防止玩家在死亡後繼續讀取。

#### 12. 作弊選單（Cheat Menu）

按下 F1 可開啟作弊選單。選單以可捲動的面板顯示所有卡牌類型（分為角色、資源、食物、建築、裝備、動物、怪物、地點、概念等分類），點擊即可在游標位置生成對應卡牌，方便助教測試與開發除錯。

## 編譯注意事項

本專案使用 CMake 建置，需要 C++17 環境。

### 建置步驟

```bash
cmake -B cmake-build-debug -DCMAKE_BUILD_TYPE=Debug
cmake --build cmake-build-debug
```

### 注意事項

- 必須使用 `-DCMAKE_BUILD_TYPE=Debug` 建置，以正確設定 `RESOURCE_DIR` 指向本地資源目錄。
- 專案使用 PTSD（Practical Tools for Simple Design）遊戲框架，透過 CMake 的 `FetchContent` 自動下載。
- 測試框架使用 Catch2 v3.7.1。

若使用新版 CMake / CLion 重新載入專案時出現以下錯誤：

```text
Compatibility with CMake < 3.5 has been removed from CMake.
```

請在 CLion 的 CMake Options 加入：

```text
-DCMAKE_POLICY_VERSION_MINIMUM=3.5
```

設定位置：

```text
File → Settings → Build, Execution, Deployment → CMake → CMake options
```

加入後請重新 Reload CMake Project。若仍無法正常編譯，可刪除 `cmake-build-debug/` 後再重新 Reload。
